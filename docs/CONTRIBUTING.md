# Documentation Contribution Standards

We welcome contributions to the LingoPulse documentation! To maintain enterprise-level quality, please follow these guidelines.

## 🖋 Tone and Style
- **Professional**: Use formal, technical English.
- **Concise**: Avoid fluff. Get straight to the implementation details.
- **Visual**: Use Mermaid diagrams for logic flows.
- **Instructional**: Always explain *why* something is implemented a certain way (ADR).

## 📁 File Naming
- Use all-caps for index files: `OVERVIEW.md`, `README.md`.
- Use snake-case or kebab-case for specific documents: `user-management.md`.
- Keep files nested within the correct category subfolder.

## 🛠 Formatting Requirements
- **Headings**: Use `#` for page title, `##` for major sections, `###` for sub-sections.
- **Code Blocks**: Always specify the language (e.g., `bash`, `php`, `json`).
- **Links**: Use relative paths (e.g., `[Link](../architecture/OVERVIEW.md)`) to allow offline browsing.
- **Alerts**: Use the following for critical info:
  - `> [!NOTE]` for extra context.
  - `> [!IMPORTANT]` for breaking changes or critical rules.

## ✅ Review Checklist
1.  Are all links working?
2.  Is the Mermaid syntax valid?
3.  Does the document align with the latest codebase?
4.  Are placeholders clearly marked?
